🪟 슬라이딩 윈도우 분할

왜 2초? 씹기 한 사이클이 0.5~1초. 2초 윈도우면 2~4번 씹는 패턴이 들어간다. 너무 짧으면 패턴 파악 불가, 너무 길면 chewing/rest 경계가 섞임.

145초 세션 → 287개 윈도우

전체: ──────────────────────────────── 145s
      [─── 2s ───]                       윈도우 1
          [─── 2s ───]                   윈도우 2  (0.5s 이동)
              [─── 2s ───]               윈도우 3
                  ...
                             [─── 2s ───] 윈도우 287

chewing: 204개 (71%)
rest:    83개  (29%)

🔢 특징 추출 (12-dim)

왜 12개? 2초 분량 ~100개 샘플을 그대로 쓰면 모델이 복잡해짐. 6개 축에서 "에너지 크기(RMS)"와 "불규칙성(Std)" 각 1개씩만 뽑으면 12개로 압축 가능. 핵심 정보는 유지됨.

📋 GT 라벨 (chewing-vision 정답지)

왜 영상 기반? IMU 데이터에는 정답이 없다. 영상에서 입 움직임(MAR)을 분석하면 "언제 씹었는지" 알 수 있다. 이 정답지를 기반으로 IMU 모델을 학습시키는 것이 핵심 전략.

chewing-vision 출력 (labels_ours.csv)

t_start  t_end   label      quality
0.0      1.0     rest       0.987
1.0      2.0     chewing    0.991
2.0      3.0     chewing    0.992
3.0      4.0     rest       0.988
...

윈도우 라벨 부여 방식

윈도우: 1.0s ~ 3.0s

  포함된 vision 라벨:
    1.0~2.0: chewing
    2.0~3.0: chewing

  chewing 비율: 2/2 = 100%
  ≥ 50% → 이 윈도우 = chewing
학습이 끝나면 vision 라벨은 더 이상 필요 없다. 배포 후엔 IMU만으로 동작.

🌲 Random Forest 학습

왜 Random Forest? 데이터가 적을 때(~300 윈도우) 딥러닝(CNN/LSTM)은 과적합됨. RF는 100개 단순 결정 트리의 투표 결과라 작은 데이터에서 안정적이고 해석 가능.

트리 1개의 판단 (단순화)

rotation_y_rms > 0.031?
├─ YES → user_accel_y_rms > 0.044?
│        ├─ YES → chewing
│        └─ NO  → rest
└─ NO  → rest

100개 트리 투표:
  72개 "chewing" → chewing 판정

Train / Test Split

287개 윈도우 (시간 순서 유지)
├─ 앞 229개: 학습 (80%)
│            모델이 정답 보고 배움
└─ 뒤  58개: 시험 (20%)
             처음 보는 데이터로 평가

왜 랜덤 섞지 않나?
→ 미래 데이터로 과거를 학습하는
  data leakage 방지
