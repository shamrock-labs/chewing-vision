# AirPods IMU 씹기 감지 — ML 파이프라인

> **목표**: 영상 없이 AirPods 센서만으로 "지금 씹고 있나?"를 판단한다.

---

## 왜 이 구조인가

직접 IMU → 씹기 감지가 어려운 이유: **정답을 모른다**.  
AirPods는 씹는 순간을 직접 알려주지 않는다.

해결책: **영상으로 정답을 만들고, 그 정답으로 IMU를 학습시킨다**.

```
[영상] → chewing-vision → 정답지(언제 씹었나)
                               ↓
[AirPods IMU] ─────────────── 학습
                               ↓
          나중엔 IMU만 있어도 예측 가능
```

학습이 끝나면 영상은 더 이상 필요 없다.

---

## 실제 데이터로 보는 파이프라인

### 입력 1 — AirPods IMU (50Hz)

```
t_rel_sec   rotation_y   user_accel_y   ...
0.3949      -0.021       0.045
0.4149      -0.034       0.078          ← 씹는 중 (Y축 흔들림)
0.4349      -0.019       0.062
0.4549       0.002       0.011          ← 쉬는 중 (거의 움직임 없음)
```

### 입력 2 — chewing-vision 정답지 (1초 단위)

```
t_start   t_end   label
0.0       1.0     rest
1.0       2.0     chewing      ← 이 1초 동안 씹고 있었음
2.0       3.0     chewing
3.0       4.0     rest
```

## 4단계 파이프라인

### Step 1. 윈도우 자르기

IMU 연속 신호를 **2초짜리 조각**으로 자른다 (0.5초씩 이동).

```
──────────────────────────────── 145초 신호
[──2초──]
   [──2초──]
      [──2초──]
               ...
→ 287개 윈도우
```

왜 2초? 씹기 한 사이클이 0.5~1초. 2초면 2~4번 씹는 패턴이 들어간다.

### Step 2. 특징 추출 (12개 숫자)

2초 윈도우를 숫자 12개로 압축한다.

```python
for axis in [rotation_x, rotation_y, rotation_z,
             user_accel_x, user_accel_y, user_accel_z]:
    RMS  = sqrt(mean(axis²))   # 에너지 크기
    Std  = std(axis)           # 얼마나 불규칙한가
```

씹을 때 → rotation_y_rms 커짐 (턱 움직임이 Y축 회전으로 나타남)  
쉴 때  → 모든 값 작음

### Step 3. 라벨 부여

윈도우가 걸쳐 있는 구간의 정답지를 본다.

```
윈도우: 1.0s ~ 3.0s
  vision 라벨: chewing(1.0~2.0), chewing(2.0~3.0)
  chewing 비율 = 100% ≥ 50% → 이 윈도우 정답 = chewing
```

### Step 4. Random Forest 학습 + 평가

```
287개 윈도우 중:
  앞 229개 → 모델에게 보여주며 학습
  뒤  58개 → 모델이 처음 보는 데이터로 시험

결과:
  F1(chewing) = 0.79   ← 기준치 0.70 통과
```

### 왜 이렇게 설계했나

파이프라인의 각 선택은 두 가지 출처에서 왔다.

**IMU 행동 인식(HAR) 분야 관행에서 가져온 것**

| 선택 | 근거 |
|------|------|
| 슬라이딩 윈도우 방식 | IMU로 행동을 분류할 때 가장 표준적인 접근. 걸음 인식, 넘어짐 감지 등 거의 모든 IMU 분류 연구가 이 구조를 씀 |
| RMS + 표준편차 피처 | "time-domain statistical features"라 부르는 조합. 딥러닝 이전 IMU 분류의 표준 — 계산이 단순하고 데이터 수백 개 수준에서도 잘 동작 |
| Random Forest | 데이터가 적을 때 SVM과 함께 IMU 분류 양대 선택지. 수백 개 샘플 규모에서 robust하다는 경험적 근거가 있음 |
| 다수결 라벨 부여 (chewing ≥ 50%) | 슬라이딩 윈도우 라벨링의 표준 방식 |

**씹기 특성에서 직접 추론한 것**

| 선택 | 추론 |
|------|------|
| 윈도우 2초 | 씹기 사이클이 0.5~1Hz → 2초 안에 2~4 사이클 포함. 1초면 사이클 1개뿐이라 노이즈에 취약 |
| stride 0.5초 | iOS에서 25샘플마다 predict하는 실시간 추론 지연과 일치. 너무 촘촘하면 중복 샘플 편향, 너무 성글면 경계 구간 누락 |
| attitude/gravity 제외 | 고개 각도에 따라 절대값이 달라져 자세 편향 발생. rotation/accel만 쓰는 게 씹기 신호에 더 안정적 |

요약: **구조는 HAR 관행**, **파라미터는 씹기 주파수(0.5~1Hz)에서 역산**. 외부 논문을 직접 인용한 건 아니고, 분야 표준을 씹기 도메인에 맞게 조정한 것.

### F1이 뭔가

정확도(Accuracy)는 "전체 중 몇 개 맞혔나"인데, 이 문제엔 잘 맞지 않는다.  
예를 들어 모든 윈도우를 무조건 "chewing"이라고 찍어도 정확도는 71%가 나온다 (chewing이 71%라서).

F1은 두 가지를 동시에 본다:

```
Precision(정밀도): 모델이 "chewing"이라 한 것 중 실제로 chewing인 비율
                   → "틀린 알람"이 얼마나 적은가

Recall(재현율):    실제 chewing 구간 중 모델이 잡아낸 비율
                   → "놓친 구간"이 얼마나 적은가

F1 = 2 × (Precision × Recall) / (Precision + Recall)
   = 둘의 조화평균 (하나가 낮으면 F1도 낮아짐)
```

구체적인 예:

```
실제:      rest  chewing  chewing  rest  chewing
예측:      rest  chewing  rest     rest  chewing
                          ↑ 놓침

Precision = 2/2 = 1.00   (chewing이라 한 2개 모두 맞음)
Recall    = 2/3 = 0.67   (실제 chewing 3개 중 2개만 잡음)
F1        = 2 × (1.00 × 0.67) / (1.00 + 0.67) = 0.80
```

기준치 **0.70**은 "놓침과 틀린 알람이 어느 정도 균형 잡혀 있다"는 최소 선.  
1.0이 만점, 0.5 이하면 찍기 수준.
