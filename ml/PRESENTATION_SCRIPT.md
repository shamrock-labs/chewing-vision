
# 발표 대본 — AirPods IMU 씹기 감지 ML 파이프라인

> PIPELINE_OVERVIEW.md 기준 / 예상 소요 시간: 10~15분

---

## 오프닝

안녕하세요. 저희 프로젝트는 **AirPods로 씹기를 감지하는 것**입니다.

조금 더 구체적으로 말씀드리면, 사람이 밥을 먹을 때 턱이 움직이잖아요. 그 움직임이 귀에 걸린 AirPods에 미세한 진동으로 전달됩니다. AirPods Pro 안에는 6축 IMU, 즉 관성 측정 장치가 내장되어 있는데, 이 센서로 그 진동 패턴을 잡아내서 "지금 씹고 있다 / 아니다"를 실시간으로 판단하는 게 목표입니다.

카메라 없이, 귀에 꽂기만 하면 씹기 횟수와 구간을 자동으로 추적할 수 있게 되는 거죠.

---

## 1. 시스템 구성

프로젝트는 크게 두 개의 파트로 나뉩니다.

첫 번째는 **chewing-vision**이라는 Python 프로젝트입니다. 이건 영상에서 입 움직임을 분석해서 "이 구간에 씹고 있다"는 정답 라벨을 자동으로 만들어주는 도구예요. 동시에 ML 모델을 학습하고 iOS에서 쓸 수 있는 형태로 내보내는 역할도 합니다.

두 번째는 **chewing-imu-collector**라는 iOS 앱입니다. iPhone으로 영상을 찍으면서 동시에 AirPods의 IMU 데이터를 50Hz로 수집하고, 학습된 모델을 탑재해서 실시간으로 씹기를 예측합니다.

이 두 파트가 서로 맞물려 돌아가는 구조입니다.

---

## 2. 전체 데이터 흐름

데이터가 어떻게 흘러가는지 순서대로 설명드릴게요.

**1단계 — 수집**

iOS 앱으로 식사 장면을 촬영하면 세 가지 파일이 만들어집니다. 영상 파일, AirPods IMU 데이터 CSV, 그리고 두 파일의 시각을 맞추기 위한 타임스탬프 앵커 JSON입니다.

**2단계 — GT 라벨 생성**

chewing-vision CLI에 영상을 넣으면, MediaPipe라는 얼굴 인식 라이브러리가 입의 열림 비율, 즉 MAR을 매 프레임에서 추출합니다. 이 값을 기반으로 1초 단위로 "씹고 있다 / 쉬고 있다"를 자동으로 라벨링합니다. 이게 우리가 사용하는 정답 데이터입니다.

**3단계 — ML 학습**

IMU 데이터와 GT 라벨을 합쳐서 Random Forest 모델을 학습합니다. 학습이 끝나면 Core ML 형식으로 변환해서 iOS 앱에 탑재합니다.

**4단계 — 실시간 추론**

앱에서 AirPods IMU 신호가 들어오면, 2초 분량의 데이터가 쌓일 때마다 0.5초 간격으로 모델이 예측을 내놓습니다. 화면에는 씹는 중인지 아닌지, 씹기 횟수가 몇 번인지 실시간으로 표시됩니다.

---

## 3. 수집 데이터 구조

데이터 파일이 어떻게 생겼는지 잠깐 보겠습니다.

**IMU CSV 파일**은 50Hz로 기록된 6축 센서 값입니다. 핵심은 `rotation_x/y/z`, 즉 각속도 세 축이에요. 씹을 때 턱이 움직이면 AirPods가 살짝 회전하는데, 이 값에 그 신호가 가장 잘 잡힙니다. `user_accel_x/y/z`도 사용하고요.

**session.json 파일**은 영상 시작 시각과 IMU 시작 시각을 맞추기 위한 기준점입니다. 두 파일의 타임스탬프 체계가 다르기 때문에, 이 파일로 오프셋을 계산해서 정렬합니다.

**labels_ours.csv**는 chewing-vision이 만들어주는 정답 라벨입니다. 1초 단위로 시작 시각, 끝 시각, 라벨이 들어있어요. 이게 없으면 ML 학습을 못 하니까 가장 중요한 파일입니다.

---

## 4. ML 파이프라인

모델이 어떻게 학습되는지 설명드릴게요.

**피처 추출**부터 시작합니다. 2초짜리 슬라이딩 윈도우를 0.5초 간격으로 밀면서, 각 윈도우 안의 6축 신호에서 RMS와 표준편차를 뽑습니다. 6축 곱하기 2개 통계량이면 총 12차원 피처 벡터가 되는 거죠.

RMS는 신호 에너지인데, 씹을 때 AirPods가 더 많이 흔들리니까 이 값이 커집니다. 표준편차는 리듬의 안정성인데, 씹기는 주기적인 운동이라서 패턴이 생기거든요.

**모델**은 Random Forest를 씁니다. 트리 20개짜리인데, 파일 크기가 20KB밖에 안 돼서 앱 용량에 부담이 없어요.

**씹기 횟수 추정**은 예측 결과에서 연속된 chewing 구간을 묶어서 bout으로 만들고, 각 bout 길이에 평균 씹기 주파수인 1.2Hz를 곱해서 계산합니다.

---

## 5. 결과 보는 법

분석 결과는 두 가지 방법으로 확인할 수 있습니다.

**HTML 리포트** — `ml/session_comparison.html`을 브라우저에서 열면 세션별 성능 비교 테이블과 시각화 이미지가 나옵니다.

시각화 이미지는 4행으로 구성되어 있어요.

첫 번째 행은 F1 점수 바차트, 씹기 비율, 추정 씹기 횟수입니다. 빨간 점선이 기준치 0.70인데, 이걸 넘어야 쓸 만한 모델입니다.

두 번째 행이 핵심인데요, 각 세션의 실제 IMU 신호 그래프입니다. 파란 선이 AirPods의 각속도 원시 신호고, 녹색 배경이 GT에서 씹고 있다고 라벨된 구간입니다. 하단에 작은 컬러 스트립이 있는데, 이게 ML이 예측한 구간이에요. 녹색 배경이랑 하단 스트립이 많이 겹칠수록 모델이 잘 맞춘 거라고 보시면 됩니다.

세 번째 행은 test set의 GT vs 예측 타임라인이고, 네 번째 행은 Confusion Matrix입니다.

**demo 영상** — `demo.mp4`를 열면 원본 영상 위에 씹기 감지 결과가 오버레이된 영상을 볼 수 있습니다. `summary.json`에는 씹기 횟수, 분당 씹기 속도 같은 수치 요약이 들어있어요.

---

## 6. 현재 성능

지금까지 수집한 세션은 3개입니다.

첫 번째 a61a4c 세션은 145초짜리인데 F1이 0.80으로 기준치를 통과했습니다. 현재 앱에 탑재된 모델이 이 세션 기준으로 학습된 거예요.

두 번째 838cje는 91초, F1이 0.871로 가장 높게 나왔습니다.

세 번째 uj2e92는 71초인데 F1이 0.286으로 낮아요. 세션이 짧아서 test set이 28개밖에 안 되다 보니 결과가 불안정한 거지, 모델 자체의 문제는 아닙니다.

세 세션을 합쳐서 학습한 combined 모델은 F1이 0.804로 안정적입니다.

---

## 7. 현재 한계와 다음 단계

솔직하게 말씀드리면, 아직 데이터가 3세션밖에 없어서 이 결과는 파이프라인 검증 수준입니다. 일반화됐다고 보기는 어렵고요.

가장 큰 문제 두 가지를 말씀드리면, 하나는 **검증 방식**입니다. 지금은 같은 세션 안에서 앞 80%로 학습하고 뒤 20%로 평가하는 time-based split을 씁니다. 하지만 앱을 실제로 쓸 때는 완전히 새로운 세션에서 잘 동작해야 하잖아요. 그걸 측정하려면 LOSO, 즉 Leave-One-Session-Out 검증이 필요합니다. 세션이 15개 이상 쌓이면 전환할 예정입니다.

두 번째는 **클래스 불균형**입니다. 현재 수집한 세션들이 씹는 구간이 65~71%로 많아서 모델이 쉬는 구간을 잘 못 잡습니다. 데이터를 수집할 때 밥 먹기 전후의 쉬는 구간을 의도적으로 더 포함시키면 해결됩니다.

목표는 15~20세션을 수집하고, LOSO 검증으로 전환하고, 전체 데이터로 모델을 재학습해서 앱에 올리는 것입니다. 그 단계가 되면 비로소 "이 모델이 새로운 세션에서도 잘 동작한다"고 말할 수 있게 됩니다.

---

## 마무리

정리하면, 저희는 AirPods IMU 신호에서 씹기를 감지하기 위해 영상 기반 GT 라벨링 + Random Forest ML 파이프라인을 구축했습니다. 현재 3세션 기준으로 combined F1 0.804를 달성했고, iOS 앱에서 실시간 추론까지 동작하는 상태입니다. 다음 목표는 15~20세션 수집 후 LOSO 검증으로 신뢰도를 높이는 것입니다.

감사합니다.
