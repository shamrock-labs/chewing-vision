# chewing-vision 발전 방향

> 현재 상태에서 최종 목표(AirPods IMU 실시간 씹기 감지)까지 가는 경로.

---

## 현재 위치

```
[영상 GT 라벨러]  ──→  [IMU ML 파이프라인]  ──→  [iOS 실시간 추론]
      ↑
  여기에 있음
  - 6세션 수집 완료
  - vision 신호 파이프라인 (jaw_open / composite)
  - LOSO 교차 검증 인프라
  - Pooled F1-chew ≈ 0.74 (composite 기준)
```

---

## 단계별 로드맵

### Phase 1 — GT 라벨 품질 확정 (지금 바로)

**목표**: 어떤 signal_mode가 더 좋은 GT 라벨을 만드는지 데이터로 결론 내기

| 작업 | 방법 | 완료 기준 |
|------|------|----------|
| v1kr2w 세션 수동 라벨링 | `python ml/annotate.py sessions/20260515T125354_v1kr2w` | labels_human.csv 완성 |
| composite vs jaw_open 정확도 비교 | annotate.py 종료 리포트 확인 | 어느 쪽이 더 human GT와 일치하는지 확인 |
| labels_ours.csv 기준 확정 | 결과에 따라 batch_reanalyze.py 재실행 | 전 세션 동일 signal_mode 적용 |

**예상 소요**: 30분 (라벨링) + 1시간 (재분석)

---

### Phase 2 — 데이터 확보 (단기)

**목표**: 세션 20개 이상 → LOSO 신뢰도 확보

| 항목 | 현재 | 목표 |
|------|------|------|
| 유효 세션 수 | 6개 | 20개 이상 |
| 수집 방법 | iOS 앱 (chewing-imu-collector) | 동일 |
| 다양성 | 단일 환경 | 음식 종류, 시간대, 고개 각도 다양화 |

수집 시 주의사항:
- 세션당 2~3분 (짧으면 test set이 너무 작아짐)
- chewing 비율 50% 근처 유지 — 쉬는 구간 의도적으로 포함
- 음식은 쌀밥/면류/샌드위치 등 씹는 질감이 다른 것들

---

### Phase 3 — ML 모델 고도화 (중기)

현재 파이프라인: 12차원 통계 피처 + Random Forest

#### 3-1. 피처 확장

```
현재 12차원:
  6축 × (RMS, std)

추가 후보:
  + 주파수 도메인 (0.8~2.5 Hz 대역 에너지)  ← 씹기 주기에 집중
  + peak interval 규칙성 (inter-peak std)
  + 크로스축 상관 (rotation_y × user_accel_y)
```

씹기는 주기적이기 때문에 주파수 도메인 피처가 특히 효과적일 가능성이 높다.

#### 3-2. 모델 교체 후보

| 모델 | 언제 고려 | 장점 |
|------|----------|------|
| Random Forest (현재) | 지금 | 데이터 소량에도 안정, 해석 가능 |
| LightGBM | 20세션 이상 | 피처 중요도 + 더 나은 일반화 |
| 1D CNN | 50세션 이상 | 시간 패턴 직접 학습, 피처 엔지니어링 불필요 |
| TCN / BiLSTM | 100세션 이상 | 긴 컨텍스트에서 씹기 패턴 학습 |

> 데이터가 적을 때 딥러닝으로 가면 과적합만 늘어난다. 세션 50개 미만에서는 RF나 LightGBM이 더 실용적.

#### 3-3. 개인화 vs 범용 모델 실험

씹기 패턴은 사람마다 꽤 다르다. 다인 데이터가 확보되면:

```
범용 모델: 모든 사람 데이터로 학습 → 처음 보는 사람에게 적용
개인화 모델: 소수 세션으로 fine-tuning → 특정 사람에게 최적화

실험: LOSO를 "세션 단위"에서 "사람 단위"로 확장
     (같은 사람 세션끼리 묶어서 LOSubjectO)
```

### Phase 4 — iOS 앱 실시간 추론 완성 (중기)

chewing-imu-collector가 이미 CoreML 추론을 지원한다. 남은 작업:

#### 4-1. 모델 재학습 + 배포

```bash
# 전체 세션으로 최종 RF 재학습
python ml/pipeline.py

# Core ML 변환
python ml/coreml_convert.py

# 앱에 ChewingClassifier.mlmodel 교체
```

#### 4-2. 실시간 피드백 UI

현재 앱: IMU 수집 + 저장  
추가 목표:
- 씹기 감지 중 실시간 카운터 표시
- 분당 씹기 속도 시각화
- 식사 완료 후 요약 리포트 (총 씹기 수, 평균 속도)

#### 4-3. 씹기 속도 피드백

씹기가 너무 빠르거나 너무 느릴 때 알림 — 천천히 씹으면 소화 개선, 포만감 향상 효과가 있다는 연구 기반.

---

### Phase 5 — 검증 및 실사용 (장기)

#### 5-1. 사람 라벨 vs 자동 라벨 대규모 비교

지금까지 수동 라벨링은 `annotate.py`로 불일치 구간만 했다. 장기적으로:
- 여러 세션 전체를 수동 라벨링
- vision 자동 라벨의 오차 패턴 파악
- 학습에서 제외할 저신뢰 구간 기준 명확화

#### 5-2. 다인 데이터 수집

지금까지는 개발팀 본인 데이터. 제품이 되려면:
- 팀원 외 사용자 데이터 수집 (동의 필요)
- 연령대, 식습관 다양화
- 개인 간 편차 측정

#### 5-3. 엣지 케이스 강화

| 케이스 | 현재 처리 | 개선 방향 |
|--------|----------|----------|
| 말하면서 씹기 | false positive 발생 | speaking 분류기 추가 |
| 고개 크게 움직임 | motion gate로 일부 억제 | IMU 자체 head motion 피처로 구분 |
| 음식 없이 껌 씹기 | 라벨 없음 | 특수 케이스로 분류 |
| 양쪽 / 한쪽 씹기 | 미지원 | AirPods L/R 신호 비교 |

## 기술 부채

| 항목 | 현황 | 우선순위 |
|------|------|---------|
| vision 라벨 신뢰도 score | quality 컬럼 있지만 ML에서 미활용 | 중 — low quality 구간 학습 제외 |
| 타임스탬프 동기화 검증 | session.json 기반이지만 drift 미확인 | 높음 — IMU-vision 어긋나면 라벨 오염 |
| bad_face 구간 처리 | 현재 학습에 포함 | 중 — 명시적으로 제외해야 함 |
| speaking 감지 | 없음 | 중 — 가장 큰 false positive 원인 |
| 세션 메타데이터 | 음식 종류, 시간대 기록 없음 | 낮음 — 추후 분석용 |
